Architecture Overview

The game uses a loop with input, update, and render. State lives in Square objects, and top-level orchestration lives in main.

Execution Flow

Each frame computes dt, handles quit events, updates living squares, respawns expired squares, applies flee behavior, and renders frame plus FPS.

State Transitions

Main transitions from Boot to Running to Stopping. death_time drives per-square transition from alive to replaced.

3 Good Patterns

  1. Descriptive function names improve readability and debugging speed.
  2. Entity data in a dataclass keeps related state together.
  3. Time-based movement with dt reduces frame-rate dependence.

2 Potential Issues

  1. resolve_square_collisions is a placeholder and can mislead expectations.
  2. No startup validation for constant relationships can cause fragile behavior if constants are changed.

Algorithm Notes

apply_flee_behavior does a pairwise neighbor scan. Smaller squares flee larger ones, larger squares chase smaller ones, then velocity is capped by size-based max speed.

Performance Score: 2 / 6

Top finding: pairwise flee checks are the first hotspot as square count increases.

  • Win: clock.tick(FPS) limits loop speed.
  • Win: motion math per square is lightweight.
  • Risk: flee behavior is near O(n^2) per frame.
  • Risk: FPS text rerender each frame adds overhead at scale.

Where to Modify Behavior
