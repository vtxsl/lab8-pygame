Runtime Architecture

This project uses a classic game loop: process input, update model state, then render. The two Python files represent two versions of the same simulation. main.py includes flee behavior, while withcol.py keeps a simpler update path.

Good Patterns
✅Constants for configuration
Window size, speeds, counts, and colors are top-level constants so behavior is easy to tune.
Basics:Edit one value, then rerun to observe direct simulation change.
Engineering Insight:Fewer hardcoded literals means less accidental mismatch across functions.
Architecture Insight:Separates policy from mechanics, a clean baseline for scaling.

✅Square dataclass model
The Square class holds state and exposes update and draw methods with clear responsibilities.
Basics:One object bundles rect, size, color, and velocity together.
Engineering Insight:Grouping related fields lowers bug chance from parameter ordering mistakes.
Architecture Insight:Model entity supports later feature growth such as health or AI flags.

✅Loop stage separation
Input, update, collision behavior, and rendering are kept in distinct phases inside main.
Basics:When something breaks, you can test one phase at a time.
Engineering Insight:Predictable stage order improves debugging and future unit test design.
Architecture Insight:Mirrors common ECS and game loop patterns used in larger engines.

Potential Issues
⚠️Collision resolver is placeholder
The function exists but currently has pass, so squares can overlap even though the name suggests collisions are handled.
Basics:Players may see objects pass through each other unexpectedly.
Engineering Insight:A silent no-op can hide defects and confuse future maintainers.
Architecture Insight:Physics contracts should be explicit, complete, or feature-flagged.
Hint: either re-enable pairwise resolution logic or rename the function to communicate current behavior.
⚠️Duplicated simulation files
main.py and withcol.py repeat most logic, which increases maintenance overhead and drift risk.
Basics:Bug fixes may be applied to one file but forgotten in the other.
Engineering Insight:Duplication multiplies review work and raises merge conflict frequency.
Architecture Insight:Shared engine module plus thin entry scripts improves extensibility.
Hint: extract shared logic into one module and keep each entry file focused on feature switches.

How To Run

Use a Python environment with pygame installed, then run one entry file:

python main.py
python withcol.py

main.py includes flee behavior. withcol.py is the simpler baseline loop.

Where To Modify Behavior

1
Simulation constants in main.py
Change NUM_SQUARES, speed limits, or size bounds to adjust difficulty and pacing.
2
apply_flee_behavior in main.py
Tune danger_radius, jitter, and clamping logic to shape AI-like escape movement.
3
resolve_square_collisions in main.py
Implement or restore pairwise collision response for true square-square bouncing.
